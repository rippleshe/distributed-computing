## 一、系统总体架构

本系统采用 **Master-Worker** 经典架构，由一个主控节点（Master）和多个计算节点（Worker）组成。

**节点角色**：
- **Master**：启动 HTTP 服务器，暴露 RESTful API，是系统的中心协调者
- **Worker**：纯客户端角色，不启动任何 HTTP 服务器，只主动调用 Master 的接口

**通信模型**：拉取式（Pull）
- Worker 主动向 Master "拉取"任务，Master 不会主动 "推送"任务给 Worker
- 这种设计使 Worker 可以部署在任意机器上，只需能访问 Master 的 HTTP 地址即可
- 多个 Worker 可以部署在同一台机器（不同端口）或不同机器上

Master 负责接收任务、拆分数据、分发子任务、汇聚结果；Worker 负责领取子任务并执行计算。系统通过 HTTP 协议进行节点间通信，使用 SQLite 作为持久化存储。

### 核心设计思想

| 设计要点 | 说明 |
|---------|------|
| **MapReduce 三阶段** | 用户只需实现 `split()` / `compute()` / `reduce()` 三个方法，框架自动完成分布式调度 |
| **动态类加载** | 用户上传 `.py` 文件，系统自动检测 Computable 子类名，Master 和 Worker 通过 `importlib` 在运行时加载并实例化用户类 |
| **CAS 乐观锁** | 所有涉及状态变更的关键操作均使用 Compare-And-Swap 防止并发冲突 |
| **内置容错调度** | Scheduler 作为 Master 的守护线程运行，Master 停止时自动结束，无需单独管理进程 |

## 二、数据库表设计

系统使用 **SQLAlchemy ORM** 定义数据库模型，底层采用 **SQLite** 文件数据库（`data/distributed.db`）。共设计三张核心表，分别存储全局任务信息、子任务分片信息和计算节点健康状态。

### 2.1 主任务表（main_task）

**用途**：存储客户端提交的全局任务信息。每当用户通过 Web UI 或 API 提交一个计算任务时，系统在此表中创建一条记录，记录任务的元信息、执行状态和最终结果。

**字段定义**：

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | 主键，任务唯一标识 |
| `task_name` | VARCHAR(200) | NOT NULL | 任务名称，用户自定义（如"计算1到100的和"） |
| `class_name` | VARCHAR(200) | NOT NULL | 用户程序类名（如"SumCalculator"），系统从上传文件中自动检测 |
| `program_path` | VARCHAR(500) | — | 程序文件在 Master 上的存储路径 |
| `program_file_name` | VARCHAR(200) | — | 原始文件名，Worker 下载时使用 |
| `input_data` | TEXT | — | 用户提交的输入参数（字符串形式） |
| `data_file_path` | VARCHAR(500) | — | 数据文件路径（可选，与 input_data 二选一） |
| `status` | VARCHAR(20) | NOT NULL, DEFAULT 'SPLITTING' | 任务状态 |
| `total_slices` | INTEGER | DEFAULT 0 | 总分片数（split 后写入） |
| `completed_slices` | INTEGER | DEFAULT 0 | 已完成分片数（实时更新） |
| `final_result` | TEXT | — | 最终汇聚结果 |
| `result_file_path` | VARCHAR(500) | — | 结果文件存储路径 |
| `created_at` | DATETIME | NOT NULL | 任务创建时间 |
| `updated_at` | DATETIME | NOT NULL | 最后更新时间（自动维护） |
| `version` | INTEGER | DEFAULT 0 | 乐观锁版本号，用于 CAS 并发控制 |

**状态流转**：

```
┌───────────┐      split() 完成      ┌──────────────┐
│ SPLITTING │ ──────────────────────→ │ IN_PROGRESS  │
└───────────┘                         └──────┬───────┘
                                             │
                          ┌──────────────────┼──────────────────┐
                          │ 用户暂停          │ 所有分片完成       │ 存在失败分片
                          ▼                  ▼                  ▼
                   ┌──────────┐       ┌─────────────┐    ┌──────────┐
                   │  PAUSED  │       │  COMPLETED  │    │  FAILED  │
                   └──────────┘       └─────────────┘    └──────────┘
                          │
                          │ 用户恢复
                          ▼
                   ┌──────────────┐
                   │ IN_PROGRESS  │
                   └──────────────┘
```

**设计说明**：
- `status` 字段采用字符串枚举，便于阅读和调试，而非数字编码
- `version` 字段用于乐观锁，在并发更新 `completed_slices` 和 `status` 时防止冲突
- `program_path` 存储绝对路径，文件保存在 `uploads/` 目录下以 UUID 子目录隔离

---

### 2.2 分任务表（task_slice）

**用途**：这是系统最核心的表，记录每个数据块的处理进度及分配情况。主任务经 `split()` 拆分后，每个分片在此表中对应一条记录。Worker 领取、执行、提交结果均围绕此表进行。

**字段定义**：

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | 主键，分片唯一标识 |
| `main_task_id` | INTEGER | NOT NULL, INDEX | 所属主任务 ID（外键逻辑关联） |
| `slice_index` | INTEGER | NOT NULL | 分片在任务中的序号（从 0 开始） |
| `input_data` | TEXT | NOT NULL | 该分片的输入数据（由 split() 生成） |
| `status` | VARCHAR(20) | NOT NULL, DEFAULT 'PENDING', INDEX | 分片状态 |
| `worker_id` | INTEGER | INDEX | 分配给哪个 Worker（领取时写入） |
| `result` | TEXT | — | 该分片的计算结果（compute() 返回值） |
| `assigned_at` | DATETIME | — | 被 Worker 领取的时间 |
| `completed_at` | DATETIME | — | 计算完成的时间 |
| `retry_count` | INTEGER | DEFAULT 0 | 已重试次数（容错机制使用） |
| `version` | INTEGER | DEFAULT 0 | 乐观锁版本号 |

**状态流转**：

```
┌─────────┐    Worker 领取     ┌──────────┐
│ PENDING │ ─────────────────→ │ ASSIGNED │
└─────────┘   (CAS 更新)       └────┬─────┘
     ▲                               │
     │                               ├──────────────────┐
     │ 容错重置                       │ 计算成功          │ 计算失败
     │ (retry_count < 3)             ▼                  ▼
     │                        ┌─────────────┐    ┌──────────┐
     └────────────────────────│  COMPLETED  │    │  FAILED  │
                              └─────────────┘    └──────────┘
```

**关键索引**：
- `main_task_id` 索引：加速按任务查询其所有分片
- `status` 索引：加速查找 PENDING 状态的待分配分片（调度核心查询）
- `worker_id` 索引：加速按 Worker 查询其负责的分片

**设计说明**：
- `status` 是调度的核心字段，Worker 通过查询 `status='PENDING'` 的记录来领取任务
- `retry_count` 配合容错调度器使用，超过 3 次重试后标记为 FAILED
- `assigned_at` 用于超时判定，调度器据此判断分片是否"僵尸"

---

### 2.3 计算节点表（compute_node）

**用途**：维护在线 Worker 的健康状态。Worker 启动时在此表注册，之后通过定时心跳维持在线状态。容错调度器根据心跳时间判断节点是否存活。

**字段定义**：

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | 主键，节点唯一标识（即 workerId） |
| `node_name` | VARCHAR(100) | NOT NULL | 节点名称（如 "worker-1"） |
| `ip_address` | VARCHAR(50) | NOT NULL | 节点 IP 地址 |
| `port` | INTEGER | NOT NULL | 节点端口号 |
| `status` | VARCHAR(20) | NOT NULL, DEFAULT 'ONLINE' | 节点状态 |
| `current_task_id` | INTEGER | — | 当前正在处理的主任务 ID |
| `current_slice_id` | INTEGER | — | 当前正在处理的分片 ID |
| `last_heartbeat` | DATETIME | NOT NULL | 最后一次心跳时间 |
| `registered_at` | DATETIME | NOT NULL | 注册时间 |
| `version` | INTEGER | DEFAULT 0 | 乐观锁版本号 |

**状态说明**：

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `ONLINE` | 空闲在线 | Worker 注册成功 或 心跳上报且无任务 |
| `BUSY` | 忙碌执行中 | 心跳上报时携带当前任务信息 |
| `OFFLINE` | 离线 | 调度器检测到心跳超时（30 秒无心跳） |

**设计说明**：
- `current_task_id` 和 `current_slice_id` 由心跳上报更新，用于监控 Worker 当前工作状态
- `last_heartbeat` 是容错判定的核心依据
- 节点记录不会自动删除，即使 Worker 下线后记录仍保留，便于追溯

### 2.4 表关系总览

```
┌─────────────────────┐       ┌─────────────────────┐       ┌─────────────────────┐
│     main_task       │       │    task_slice        │       │   compute_node      │
│  (主任务表)          │       │  (分任务表)          │       │  (计算节点表)        │
├─────────────────────┤       ├─────────────────────┤       ├─────────────────────┤
│ id (PK)             │◄──┐   │ id (PK)             │       │ id (PK)             │
│ task_name           │   │   │ main_task_id (FK) ──┘   │   │ node_name           │
│ class_name          │   │   │ slice_index             │   │ ip_address          │
│ program_path        │   │   │ input_data              │   │ port                │
│ status              │   │   │ status                  │   │ status              │
│ total_slices        │   │   │ worker_id ──────────────│──►│ current_task_id     │
│ completed_slices    │   │   │ result                  │   │ current_slice_id    │
│ final_result        │   │   │ retry_count             │   │ last_heartbeat      │
│ version             │   │   │ version                 │   │ version             │
└─────────────────────┘   │   └─────────────────────────┘   └─────────────────────┘
                          │
                    一对多关系
                  (1 : N)
```

**关系说明**：
- `main_task` 与 `task_slice`：一对多关系。一个主任务经 split 后产生 N 个分片
- `task_slice.worker_id` 与 `compute_node.id`：逻辑外键关联，记录分片被哪个 Worker 领取
- 三张表均包含 `version` 字段，用于乐观锁并发控制

## 三、通讯基础框架

### 3.1 主控节点（Master）

**技术选型**：基于 **Flask** 轻量级 Web 框架，接收 HTTP 请求。

**职责范围**：
- 接收客户端提交的任务请求
- 自动检测用户程序中的 Computable 子类（无需手动指定类名）
- 动态加载用户上传的 Python 程序
- 执行 `split()` 将任务拆分为多个分片
- 响应 Worker 的注册、心跳、任务领取请求
- 接收 Worker 提交的分片结果
- 执行 `reduce()` 汇聚最终结果
- 提供 Web UI 管理界面
- 内置容错调度器守护线程（自动巡检，无需单独管理）

**启动方式**：
```
uv run python master.py --port 8080
```

**模块结构**：

```
master.py
├── Flask Web Server              ← HTTP 服务层
│   ├── /                         ← Web UI 首页（templates/index.html）
│   ├── /api/tasks                ← 任务 CRUD 接口
│   ├── /api/workers              ← Worker 管理接口
│   └── /api/tasks/<id>/slices/<id>/result  ← 结果提交接口
├── load_program()                ← 动态类加载器
├── detect_computable_class()     ← 自动检测 Computable 子类名
├── TaskService                   ← 任务业务逻辑
│   ├── submit_task()             ← 接收任务 → 自动检测类 → split() → 创建分片
│   ├── get_next_slice()          ← 查找 PENDING 分片 → CAS 更新为 ASSIGNED
│   └── submit_slice_result()     ← 更新分片结果
├── FaultToleranceScheduler       ← 容错调度器（守护线程）
    ├── check_offline_workers()   ← 检测离线 Worker
    ├── check_timeout_slices()    ← 重置超时分片
    ├── check_task_failure()      ← 传播任务失败状态
    └── check_and_reduce_tasks()  ← 检查已完成任务并执行 reduce
```

**交互方式**：页面（Web 浏览器）或客户端程序直接发送 HTTP 请求。

**类名自动检测**：用户上传 `.py` 文件后，系统自动扫描模块中继承 `Computable` 的类。如果恰好有一个，直接使用；如果有多个或没有，返回错误提示。用户无需手动填写类名。

---

### 3.2 计算节点（Worker）

**技术选型**：纯 Python 实现，使用 `requests` 库与 Master 进行 HTTP 通信。

**重要**：Worker 不启动任何 HTTP 服务器，不监听端口，不接收请求。它是纯客户端角色，所有通信都是 Worker 主动调用 Master 的 API。

**职责范围**：
- 启动时调用 Master 的注册接口，获取 `workerId`
- 定时调用心跳接口，报告自身健康状态
- 轮询 Master 的领取任务接口，获取待处理的分片
- 从 Master 下载用户程序文件
- 动态加载并执行 `compute()` 方法
- 调用 Master 的提交结果接口，返回计算结果

**通信模型（Pull 拉取式）**：

```
┌─────────────────────┐                    ┌─────────────────────┐
│       Master        │                    │       Worker        │
│                     │                    │                     │
│  HTTP 服务器         │◄── Worker 主动 ──  │  纯客户端            │
│  暴露 RESTful API   │    调用接口         │  不接收任何请求       │
│                     │                    │                     │
│  /api/workers/reg   │◄── 注册            │  requests.post()    │
│  /api/workers/hb    │◄── 心跳            │  requests.post()    │
│  /api/workers/next  │◄── 领取任务        │  requests.get()     │
│  /api/tasks/program │◄── 下载程序        │  requests.get()     │
│  /api/tasks/result  │◄── 提交结果        │  requests.post()    │
└─────────────────────┘                    └─────────────────────┘
```

**启动方式**：
```
uv run python worker.py --port 8081 --name worker-1 --master http://localhost:8080
```

注意：`--port` 参数仅在注册时上报给 Master 用于标识，Worker 本身不监听该端口。

**模块结构**：

```
worker.py
└── WorkerService（纯客户端，不启动 HTTP 服务器）
    ├── register()            ← 调用 Master 注册接口
    ├── send_heartbeat()      ← 调用心跳接口（每 3 秒）
    ├── request_task()        ← 调用领取任务接口（每 1 秒）
    ├── execute_task()        ← 异步执行（新线程）
    │   ├── download_program()← 调用下载程序接口
    │   ├── load_program()    ← importlib 动态加载
    │   └── compute()         ← 调用用户计算方法
    ├── submit_result()       ← 调用提交结果接口
    └── run()                 ← 主循环入口
```

## 五、分块实现

本系统的核心计算模型仿照 MapReduce 设计，分为三个阶段：**Split（分块）→ Compute（计算）→ Reduce（汇聚）**。用户只需继承 `Computable` 基类并实现这三个方法，框架自动完成分布式调度。

### 5.1 分任务程序的实现（Split）

**执行位置**：Master 端

**触发时机**：客户端提交任务后，Master 立即执行

**功能描述**：将用户提交的原始输入数据切分为多个独立的数据块（分片），每个分片将被分配给一个 Worker 独立计算。

**执行流程**：

```
用户提交任务
    │
    ▼
Master 接收程序文件和输入数据
    │
    ▼
importlib 动态加载用户 .py 文件
    │
    ▼
实例化用户类（继承 Computable）
    │
    ▼
调用 split(input_data)
    │
    ▼
返回 List[str]（分片列表）
    │
    ▼
为每个分片创建 task_slice 记录
    │
    ▼
所有分片初始状态设为 PENDING
```

**分片策略示例**：

以累加计算为例，输入 `"1,100"`，按每 25 个数切分：

```
输入: "1,100"
    │
    ▼ split()
    │
    ├── 分片 0: "1,25"     ← PENDING
    ├── 分片 1: "26,50"    ← PENDING
    ├── 分片 2: "51,75"    ← PENDING
    └── 分片 3: "76,100"   ← PENDING
```

以词频统计为例，输入为文本文件内容，按每 10 行切分：

```
输入: 50 行文本
    │
    ▼ split()
    │
    ├── 分片 0: 第 1~10 行    ← PENDING
    ├── 分片 1: 第 11~20 行   ← PENDING
    ├── 分片 2: 第 21~30 行   ← PENDING
    ├── 分片 3: 第 31~40 行   ← PENDING
    └── 分片 4: 第 41~50 行   ← PENDING
```

**设计要点**：
- 分片策略由用户自行决定，框架不干预
- 分片数量直接影响并行度，建议根据数据规模和 Worker 数量合理划分
- 每个分片必须是独立可计算的，不能依赖其他分片的结果

---

### 5.2 计算程序的实现（Compute）

**执行位置**：Worker 端

**触发时机**：Worker 从 Master 领取到分片后执行

**功能描述**：处理单个分片的数据，返回该分片的计算结果。多个 Worker 并行执行各自的 `compute()`，互不干扰。

**执行流程**：

```
Worker 轮询 Master 获取分片
    │
    ▼
领取成功（CAS 更新 PENDING → ASSIGNED）
    │
    ▼
从 Master 下载程序文件到本地
    │
    ▼
importlib 动态加载用户类
    │
    ▼
调用 compute(slice_data)
    │
    ▼
获得计算结果（字符串）
    │
    ▼
提交结果给 Master（POST 请求）
    │
    ▼
Master 更新 task_slice（status=COMPLETED, result=...）
```

**并行计算示意**：

```
                    ┌─────────┐
                    │  Master │
                    └────┬────┘
                         │ 分发分片
          ┌──────────────┼──────────────┐
          │              │              │
          ▼              ▼              ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐
    │ Worker-1 │   │ Worker-2 │   │ Worker-3 │
    │          │   │          │   │          │
    │ compute  │   │ compute  │   │ compute  │
    │("1,25")  │   │("26,50") │   │("51,75") │
    │          │   │          │   │          │
    │ result:  │   │ result:  │   │ result:  │
    │  "325"   │   │  "950"   │   │  "1575"  │
    └────┬─────┘   └────┬─────┘   └────┬─────┘
         │              │              │
         └──────────────┼──────────────┘
                        │ 提交结果
                        ▼
                    ┌─────────┐
                    │  Master │
                    └─────────┘
```

---

### 5.3 汇聚程序的实现（Reduce）

**执行位置**：Master 端

**触发时机**：当某个任务的所有分片均完成（COMPLETED）时自动触发

**功能描述**：将所有分片的计算结果合并为最终结果。这是 MapReduce 中 Reduce 阶段的简化实现。

**执行流程**：

```
Worker 提交分片结果
    │
    ▼
Master 更新 task_slice（status=COMPLETED）
    │
    ▼
CAS 更新 main_task.completed_slices
    │
    ▼
检查：completed_slices == total_slices ?
    │
    ├── 否 → 返回，等待更多分片完成
    │
    └── 是 → 执行 reduce()
                │
                ▼
         收集所有分片的 result
                │
                ▼
         调用 program.reduce(results)
                │
                ▼
         保存最终结果到文件
                │
                ▼
         更新 main_task（status=COMPLETED, final_result=...）
```

**汇聚示意**：

```
分片结果收集：
    ┌──────────┬──────────┬──────────┬──────────┐
    │ "325"    │ "950"    │ "1575"   │ "2200"   │
    └────┬─────┴────┬─────┴────┬─────┴────┬─────┘
         │          │          │          │
         └──────────┴──────┬───┴──────────┘
                           │
                           ▼ reduce()
                           │
                    325 + 950 + 1575 + 2200
                           │
                           ▼
                       "5050"
```

**并发安全**：
- `check_and_reduce()` 使用 CAS 更新 `completed_slices`，确保只有一个线程能成功触发 `reduce()`
- 即使多个 Worker 同时提交结果，`reduce()` 也只会执行一次

## 六、计算节点运行框架

### 6.1 心跳与任务状态更新

Worker 采用 **轮询 + 心跳** 双机制与 Master 保持通信：

```
┌─────────────────────────────────────────────────────────────┐
│                    Worker 主循环                              │
│                                                             │
│  while True:                                                │
│      │                                                      │
│      ├── 每 3 秒 ──→ 发送心跳（上报状态）                     │
│      │                                                      │
│      ├── 每 1 秒 ──→ 请求任务（领取分片）                     │
│      │                                                      │
│      └── 休眠 500ms ──→ 循环                                 │
│                                                             │
│  领取到任务后：                                               │
│      └── 新线程 ──→ execute_task()（异步执行，不阻塞主循环）   │
└─────────────────────────────────────────────────────────────┘
```

---

### 6.2 心跳信息

**发送频率**：每 3 秒一次

**心跳内容**：

```
┌─────────────────────────────────────────────────────────┐
│  心跳报文                                                │
│  ┌───────────────┬────────────────────────────────────┐  │
│  │ 字段           │ 说明                               │  │
│  ├───────────────┼────────────────────────────────────┤  │
│  │ status        │ ONLINE（空闲）/ BUSY（执行中）       │  │
│  │ currentTaskId │ 当前正在处理的任务 ID（可选）        │  │
│  │ currentSliceId│ 当前正在处理的分片 ID（可选）        │  │
│  └───────────────┴────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

**Master 处理逻辑**：

```
收到心跳
    │
    ▼
更新 compute_node 表：
    ├── status = 上报的状态
    ├── current_task_id = 上报的任务 ID
    ├── current_slice_id = 上报的分片 ID
    ├── last_heartbeat = 当前时间
    └── version += 1
```

**容错联动**：如果 Worker 在执行任务过程中崩溃，心跳停止，调度器会在 30 秒后将其标记为 OFFLINE，并在 60 秒后重置其负责的分片。

---

### 6.3 任务状态同步

任务状态在多个维度上保持同步：

```
┌─────────────────────────────────────────────────────────────────┐
│                    状态同步机制                                    │
│                                                                 │
│  ┌─────────────┐    分片完成时     ┌──────────────────────────┐  │
│  │ task_slice  │ ──────────────→  │ main_task               │  │
│  │ COMPLETED   │   更新计数器      │ completed_slices += 1   │  │
│  └─────────────┘                  └──────────────────────────┘  │
│                                                                 │
│  ┌─────────────┐    心跳上报时     ┌──────────────────────────┐  │
│  │ Worker 状态 │ ──────────────→  │ compute_node            │  │
│  │ ONLINE/BUSY │   更新节点记录    │ status, last_heartbeat  │  │
│  └─────────────┘                  └──────────────────────────┘  │
│                                                                 │
│  ┌─────────────┐    巡检定时器     ┌──────────────────────────┐  │
│  │ 超时检测    │ ──────────────→  │ task_slice / compute_node│  │
│  │ Scheduler   │   重置/标记离线   │ status 更新              │  │
│  └─────────────┘                  └──────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 6.4 进度查询

客户端可随时查询任务执行进度，Master 从数据库实时读取：

```
GET /api/tasks/<taskId>

响应中的关键进度字段：
{
  "totalSlices": 4,           ← 总分片数
  "completedSlices": 2,       ← 已完成分片数
  "progress": 50.0,           ← 完成百分比
  "status": "IN_PROGRESS"     ← 任务状态
}

进度计算公式：
progress = (completed_slices / total_slices) × 100%
```

**Web UI 自动刷新**：前端页面每 5 秒自动轮询任务列表和 Worker 状态，实现实时监控。

## 七、容错机制

容错是分布式系统的核心挑战。本系统通过 Master 内置的 **容错调度器守护线程**，定时巡检数据库，处理各类异常情况。

### 7.1 定时巡检逻辑

Scheduler 作为 **Master 进程内的守护线程** 运行，与 Master 共享同一个 SQLite 数据库。每 **15 秒**执行一次完整巡检。

**线程集成设计**：

```
┌─────────────────────────────────────────────────────────────────┐
│  Master 进程                                                     │
│                                                                 │
│  ┌─────────────────────────────┐                                │
│  │  Flask 主线程                │                                │
│  │  处理 HTTP 请求              │                                │
│  └─────────────────────────────┘                                │
│                                                                 │
│  ┌─────────────────────────────┐                                │
│  │  容错调度器守护线程（daemon） │                                │
│  │  每 15 秒巡检一次            │                                │
│  │  - 检测离线 Worker           │                                │
│  │  - 重置超时分片              │                                │
│  │  - 传播任务失败状态          │                                │
│  └─────────────────────────────┘                                │
│                                                                 │
│  两个线程通过 SQLite 数据库协调，互不阻塞                          │
│  Master 停止时，守护线程自动结束                                   │
└─────────────────────────────────────────────────────────────────┘
```

**巡检流程**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    容错调度器巡检流程                              │
│                                                                 │
│  每 15 秒执行一次                                                │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ ① 检查 Worker 心跳（check_offline_workers）               │   │
│  │                                                          │   │
│  │   条件：last_heartbeat < NOW() - 30秒                     │   │
│  │   动作：status → OFFLINE                                  │   │
│  └──────────────────────────────────────────────────────────┘   │
│                          │                                       │
│                          ▼                                       │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ ② 检查分片超时（check_timeout_slices）                    │   │
│  │                                                          │   │
│  │   条件：status=ASSIGNED 且 assigned_at < NOW() - 60秒     │   │
│  │   动作：                                                  │   │
│  │     retry_count < 3  → 重置为 PENDING，retry_count++     │   │
│  │     retry_count >= 3 → 标记为 FAILED                     │   │
│  └──────────────────────────────────────────────────────────┘   │
│                          │                                       │
│                          ▼                                       │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ ③ 检查任务失败传播（check_task_failure）                   │   │
│  │                                                          │   │
│  │   条件：任务的所有分片都已终态 且 存在 FAILED 分片          │   │
│  │   动作：main_task.status → FAILED                         │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 超时判定算法

超时判定基于 **时间差值** 算法，核心思想是将记录的更新时间与当前时间比较：

```
┌─────────────────────────────────────────────────────────────────┐
│                    超时判定算法                                    │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Worker 心跳超时                                           │   │
│  │                                                          │   │
│  │  判定公式：                                               │   │
│  │    elapsed = NOW() - last_heartbeat                      │   │
│  │    is_timeout = elapsed > 30 秒                          │   │
│  │                                                          │   │
│  │  SQL 表达：                                               │   │
│  │    WHERE last_heartbeat < NOW() - INTERVAL 30 SECOND     │   │
│  │                                                          │   │
│  │  超时处理：                                               │   │
│  │    UPDATE compute_node SET status = 'OFFLINE'            │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 分片分配超时（僵尸任务检测）                                │   │
│  │                                                          │   │
│  │  判定公式：                                               │   │
│  │    elapsed = NOW() - assigned_at                         │   │
│  │    is_timeout = elapsed > 60 秒                          │   │
│  │                                                          │   │
│  │  SQL 表达：                                               │   │
│  │    WHERE status = 'ASSIGNED'                             │   │
│  │      AND assigned_at < NOW() - INTERVAL 60 SECOND        │   │
│  │                                                          │   │
│  │  超时处理：                                               │   │
│  │    retry_count < 3  → 重置为 PENDING                     │   │
│  │    retry_count >= 3 → 标记为 FAILED                      │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

**超时参数配置**：

| 参数 | 值 | 说明 |
|------|-----|------|
| `HEARTBEAT_TIMEOUT` | 30 秒 | Worker 无心跳超过此时间视为离线 |
| `SLICE_TIMEOUT` | 60 秒 | 分片被领取后超过此时间未完成视为超时 |
| `MAX_RETRY` | 3 次 | 分片最大重试次数，超过后标记为 FAILED |

---

### 7.3 数据状态重置

当检测到超时分片后，调度器执行状态重置，将失效任务标记回 PENDING，使其他 Worker 有机会再次领取。

**重置逻辑**：

```
超时分片（ASSIGNED 超过 60 秒）
    │
    ▼
检查 retry_count
    │
    ├── retry_count < 3
    │       │
    │       ▼
    │   ┌─────────────────────────────────────────┐
    │   │ 重置操作：                                │
    │   │   status      → PENDING                  │
    │   │   worker_id   → NULL                     │
    │   │   retry_count += 1                       │
    │   │   version     += 1                       │
    │   └─────────────────────────────────────────┘
    │       │
    │       ▼
    │   其他 Worker 可以重新领取此分片
    │
    └── retry_count >= 3
            │
            ▼
        ┌─────────────────────────────────────────┐
        │ 终止操作：                                │
        │   status → FAILED                        │
        │   version += 1                           │
        └─────────────────────────────────────────┘
            │
            ▼
        该分片不再重试，最终可能导致主任务 FAILED
```

**状态重置的连锁反应**：

```
分片重置为 PENDING
    │
    ▼
其他 Worker 轮询时可领取
    │
    ▼
重新执行 compute()
    │
    ▼
提交结果 → 更新 completed_slices
    │
    ▼
如果所有分片最终完成 → 触发 reduce()
```

### 7.4 并发控制：乐观锁机制

在分布式环境中，多个 Worker 可能同时抢占同一个 PENDING 分片。本系统采用 **CAS（Compare-And-Swap）乐观锁** 解决此问题。

**问题场景**：

```
时间线：
    T1          T2          T3
    │           │           │
    ▼           ▼           ▼
Worker-A     Worker-B     两个 Worker 同时
查询到       查询到       领取同一个分片
分片 #5      分片 #5      → 数据冲突！
(PENDING)    (PENDING)
```

**CAS 解决方案**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    CAS 乐观锁流程                                 │
│                                                                 │
│  Worker-A                                    Worker-B           │
│     │                                           │               │
│     ▼                                           ▼               │
│  ① SELECT * FROM task_slice                    ① 同样的查询      │
│     WHERE status = 'PENDING'                      │               │
│     → 获取分片 #5, version = 3                    │               │
│     │                                           │               │
│     ▼                                           ▼               │
│  ② UPDATE task_slice                           ② 同样的更新      │
│     SET status = 'ASSIGNED'                       │               │
│     WHERE id = 5                                  │               │
│       AND status = 'PENDING'                      │               │
│       AND version = 3        ← CAS 条件           │               │
│     │                                           │               │
│     ▼                                           ▼               │
│  affected = 1 (成功)                           affected = 0     │
│  version → 4                                   (CAS 失败！)     │
│  → Worker-A 获得分片                           → 无可用任务      │
│                                                                 │
│  关键：CAS 操作是原子的，数据库保证只有一个 Worker 能成功更新      │
└─────────────────────────────────────────────────────────────────┘
```

**CAS 在系统中的应用点**：

| 操作 | CAS 条件 | 说明 |
|------|---------|------|
| 领取分片 | `status='PENDING' AND version=?` | 防止多个 Worker 抢占同一分片 |
| 提交结果 | `version=?` | 防止重复提交 |
| 更新进度 | `version=?` | 防止并发更新 completed_slices |
| 暂停/恢复 | `version=?` | 防止并发状态变更 |

**为什么选择乐观锁而非悲观锁**：

```
┌─────────────────────────────────────────────────────────────────┐
│  乐观锁 vs 悲观锁                                                │
│                                                                 │
│  悲观锁（SELECT ... FOR UPDATE）：                                │
│    优点：实现简单，不会冲突                                       │
│    缺点：锁粒度大，吞吐量低                                       │
│    适用：冲突频繁的场景                                           │
│                                                                 │
│  乐观锁（CAS version）：                                         │
│    优点：无锁开销，吞吐量高                                       │
│    缺点：冲突时需要重试                                           │
│    适用：冲突较少的场景（本系统的情况）                              │
│                                                                 │
│  本系统选择乐观锁的原因：                                          │
│    1. Worker 数量有限，同时抢占同一分片的概率较低                    │
│    2. SQLite 不支持 SELECT FOR UPDATE SKIP LOCKED                 │
│    3. CAS 失败时 Worker 会自动轮询下一个分片，无需重试              │
└─────────────────────────────────────────────────────────────────┘
```

## 八、用户程序接口规范

用户通过继承 `Computable` 基类来编写分布式计算程序。该基类定义了三个抽象方法，对应 MapReduce 的三个阶段。

### 接口定义

```
┌─────────────────────────────────────────────────────────────────┐
│  Computable（抽象基类）                                          │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  split(input_data: str) → List[str]                       │  │
│  │                                                           │  │
│  │  执行位置：Master 端                                       │  │
│  │  功能：将输入数据切分为多个分片                              │  │
│  │  参数：原始输入数据（参数字符串或文件内容）                    │  │
│  │  返回：分片列表，每个元素是一个分片的输入数据                  │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  compute(slice_data: str) → str                           │  │
│  │                                                           │  │
│  │  执行位置：Worker 端                                       │  │
│  │  功能：处理单个分片，返回计算结果                            │  │
│  │  参数：单个分片的输入数据                                   │  │
│  │  返回：该分片的计算结果（字符串）                            │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  reduce(results: List[str]) → str                         │  │
│  │                                                           │  │
│  │  执行位置：Master 端                                       │  │
│  │  功能：将所有分片结果合并为最终结果                          │  │
│  │  参数：所有分片的计算结果列表                               │  │
│  │  返回：最终结果（字符串）                                   │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 已实现的示例程序

**示例一：累加计算（SumCalculator）**

```
输入: "1,100"
    │
    ▼ split()
    ├── "1,25"
    ├── "26,50"
    ├── "51,75"
    └── "76,100"
    │
    ▼ compute() × 4 个 Worker 并行
    ├── "325"
    ├── "950"
    ├── "1575"
    └── "2200"
    │
    ▼ reduce()
    "5050"
```

**示例二：词频统计（WordCount）**

```
输入: 50 行文本文件

    ▼ split()（按每 10 行切分）
    ├── 第 1~10 行
    ├── 第 11~20 行
    ├── 第 21~30 行
    ├── 第 31~40 行
    └── 第 41~50 行
    │
    ▼ compute() × 5 个 Worker 并行
    ├── "hello:5\nworld:4\n..."
    ├── "hello:6\nworld:5\n..."
    ├── "hello:4\nworld:5\n..."
    ├── "hello:3\nworld:4\n..."
    └── "hello:4\nworld:3\n..."
    │
    ▼ reduce()（合并词频，降序排列）
    "hello:22\nworld:21\n..."
```

## 九、系统部署与启动

### 依赖环境

```
Python >= 3.10
Flask >= 3.0
requests >= 2.31
SQLAlchemy >= 2.0
```

### 启动方式

**方式一：一键启动（推荐）**

```
uv run python start.py
```

此脚本自动完成：
1. 初始化数据库（创建表结构）
2. 启动 Master（端口 8080，自动启动容错调度器守护线程）
3. 启动 10 个 Worker（端口 8081~8090）

**方式二：分别启动**

```
# 终端 1：启动 Master（自动包含容错调度器）
uv run python master.py --port 8080

# 终端 2~N：启动 Worker
uv run python worker.py --port 8081 --name worker-1
uv run python worker.py --port 8082 --name worker-2
```

**注意**：容错调度器已集成到 Master 进程内（守护线程），无需单独启动 scheduler.py。Master 停止时调度器自动结束。
